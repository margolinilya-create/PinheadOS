## Выводы для модели ERP

1. **Нанесение — отдельная сущность** (`erp_item_prints`): позиция может иметь
   N нанесений, каждое со своей техникой, зоной, размером, Pantone, макетом.
   Сейчас это плоские галочки — маршрут строится верно, но параметров нет.
2. **Размерная сетка** (`erp_item_sizes` или JSONB): цвет × размер → qty.
   Итог qty должен сходиться с суммой сетки.
3. **Материалы по ролям**: основное полотно / отделочное / подклад / фурнитура
   + цвет + поставщик — расширение erp_materials (role, color, supplier).
4. **Упаковка и маркировка на заказе**: packaging (none/bopp/zip/other+text),
   stickers (none/blank/other+text), no_chestny_znak boolean,
   бирки (размерник/составник + способ притачивания).
5. **Галерея вложений с подписями**: kind='note' + caption — вместо
   одного превью (стр. 4 = фото-карточки).
6. Форма создания становится **многосекционной** (как ТЗ): Заказ → Позиция
   (модель, сетка размеров, полотна) → Нанесения (повторяемые) → Упаковка/бирки.
   ТЗ-PDF в перспективе генерируется из этих полей (PrintPreview уже есть
   в Order Studio — может стать рендером ТЗ).

## Решение заказчика (2026-07-17)

- Создание ТЗ — в разделе ТЗ (Order Studio); после создания заказ попадает
  в производство. Мост и генерация ТЗ-PDF — позже, в своём разделе.
- Сейчас горит производство: делаем пункты 1 и 2 —
  (1) БД + форма ERP: размерная сетка, нанесения-сущность, упаковка/стикеры/ЧЗ,
      материалы по ролям;
  (2) карточка заказа показывает всё это цехам.
