# kontora24 → PinheadOS ERP: анализ конфликтов и план внедрения

> Принцип: **логика заказов PinheadOS приоритетна.** Механики kontora24
> адаптируются под нашу модель, а не наоборот.
> Разбор донора: docs/erp/kontora24-analysis.md.

## 0. Корневое различие моделей

| | kontora24 | PinheadOS ERP (приоритет) |
|---|---|---|
| Единица движения | **Заказ целиком**, один линейный `status` (19 статусов) | **Позиция × этапы-граф** (`erp_item_stages` с `depends_on`), статус у каждого этапа |
| Параллельность | Костыль: 2 «трека» для 3D-паков | Нативно: любые параллельные ветки нанесения |
| Маршрут | Вычисляется на лету (`getOrderRoute` по флагам) | **Материализован в БД** при создании (buildRoute) |
| Прогресс заказа | = его статус | Вычисляется из этапов (готово X из Y) |
| Движение | Заказ прыгает по колонкам | Цех завершает свой этап → зависимые открываются |

Вывод: **всё, что у kontora24 завязано на «один статус заказа», переносится
только идейно** (UX, guard'ы, история), с переписыванием на этапную модель.

## 1. Таблица конфликтов

### 🔴 Конфликтуют с нашей моделью — только адаптация

| Механика kontora24 | Конфликт | Вердикт |
|---|---|---|
| Канбан «колонка = статус заказа», DnD двигает заказ | У нас позиция может быть в 2 цехах одновременно (ветки); этап привязан к цеху — «перетащить в другой цех» бессмысленен | **Адаптируем**: канбан = обзор «где работа сейчас»: колонка = цех, карточка = этап позиции. DnD не «куда угодно», а только в след. колонку по графу = завершить этап (ready→in_progress→done). Параллельные ветки — карточка в каждой колонке своей ветки |
| `updateOrderStatus(from,to)` — переход заказа | У нас нет статуса-источника; есть setStageStatus | **Идейно**: переносим guard-слои (маршрут → RBAC → мягкий completion) внутрь setStageStatus; isRollback/force = админ-откат этапа (у нас частично есть через «Брак») |
| `getOrderRoute` на лету + off-route баннер | Наш маршрут материализован: смена флагов заказа не двигает этапы задним числом (история цеховых работ не должна «уезжать») | **Не переносим** пересчёт на лету. Вместо off-route баннера — действие «Пересобрать маршрут» в карточке (admin), с сохранением done-этапов |
| Единый линейный Stepper заказа | У нас граф | **Адаптируем**: stepper по позиции = сортировка sort_order, параллельные ветки в одной «ячейке» лесенкой (без SVG-развилок kontora24 — у нас проще) |
| `production_logs` (много записей на этап) vs наш qty_done одним числом | Пересечение с erp_stage_events | **Наше приоритетно**: события уже пишутся; qty_done делаем накопительным (sum по событиям), отдельную таблицу логов не заводим |
| Финансы в заказе (price_final, маскирующее view) | Финансы у нас заморожены решением заказчика | **Не переносим** сейчас; паттерн view запомнили на потом |
| @dnd-kit (новая зависимость) | Правило проекта: не добавлять npm-зависимости без обсуждения; наш KanbanBoard — нативный HTML5 DnD + клавиатура, 0 deps | **Рекомендация: наш движок** из Order Studio, механики (блокировка колонок, время-в-статусе) — от kontora24 |
| 19 статусов, sample workflow (образец→цветопроба→тираж) | У нас образцы = тип производства с маршрутом через эксперим. цех | **Не переносим** статусы; sample workflow уже покрыт нашей моделью |

### 🟡 Совместимы после мелкой адаптации

| Механика | Адаптация |
|---|---|
| `isStageAllowed` (includes в массиве) | У нас = `isStageReady` (граф+материалы) — **уже есть**, переиспользуем для блокировки колонок канбана |
| `computeIncoming` («не больше, чем пришло») | Их «предыдущий этап в массиве» → наш `depends_on`: incoming = min(qty_done − qty_rework) по зависимостям. Severity warning, не блок — как у них |
| Multi-step прогон с автооткатом | У нас = завершение цепочки этапов одной позиции; öткат через возврат статусов. Упрощается: шагов меньше |
| Время-в-статусе | У нас = now − started_at этапа (или updated_at для waiting). Прямо ложится |
| Роли уведомлений NOTIFY_ROLES | Их роли ≠ наши; маппим на цех: событие «этап стал ready в цехе X» → уведомить сотрудников цеха X |
| Клиенты (`findOrCreateClientByName`) | У нас клиента нет вообще → **аддитивно**: erp_clients + поле client_id (nullable), старые заказы не трогаем |

### 🟢 Без конфликтов — переносим как есть

- EditableField-паттерн (инлайн-правки в карточке)
- Оптимистичный `setOrder(prev=>({...prev,...patch}))` (у нас уже так в сторе)
- Комментарии-чат заказа с realtime (новая таблица erp_order_comments)
- Audit-триггер полей заказа (erp_order_audit) + единая History-лента
- Realtime: уникальные имена каналов + debounce 500ms
- SavedFilters (JSONB в БД, RLS «только свои»)
- DateRangeFilter с пресетами; переключатель видов список/канбан
- Drop-zone превью с Ctrl+V; плитки типов; пресеты чипами; error-summary
- Двухступенчатый сабмит (Zod → preSubmit → ConfirmDialog)
- Табы desktop / dropdown mobile

## 2. План внедрения (волны)

### Волна 1 — Канбан цехов + движение (ядро)
1. `ErpKanban`: колонки = 7 рабочих цехов, карточка = этап позиции
   (наш HTML5-движок из Order Studio, механики kontora24)
2. Карточка канбана: дедлайн-точка, время-в-этапе, qty, заказ/позиция
3. DnD: перетаскивание карточки в колонку = недопустимо, кроме жеста
   «в done-зону» (завершить); блокировка серым для не-ready
4. Guard-слои в setStageStatus (RBAC по цеху сотрудника — мягкий)
5. Переключатель видов на /board: таблица (текущий план) / канбан

### Волна 2 — Карточка заказа
1. InlineEdit-компонент (клик→input→Enter) → номер, срок, менеджер, приоритет
2. Stepper позиции с tooltip «когда/кто» из erp_stage_events
3. erp_order_audit (триггер) + объединённая лента История (статусы+правки)
4. erp_order_comments + чат в карточке (realtime)
5. computeIncoming: предупреждение «швейке пришло только 95 из 100»

### Волна 3 — Создание заказа
1. erp_clients + findOrCreateClientByName, поле «Клиент» в форме
2. Тип производства плитками, error-summary, scrollToFirstError
3. Drop-zone превью позиции (Storage bucket erp-attachments) + Ctrl+V
4. Двухступенчатый сабмит (бизнес-проверки: дубли позиций, срок < сегодня)

### Волна 4 — Живость
1. Realtime-подписка на erp_item_stages/erp_orders (debounce) — доска
   и очереди обновляются сами
2. Уведомления цеху: «этап готов к работе» (звук + toast)
3. SavedFilters + DateRangeFilter на заказах/плане

## 3. Решения, которые нужно подтвердить

1. **DnD-движок**: наш HTML5 (0 зависимостей, рекомендую) или @dnd-kit (их код
   переносится дословно, но новая зависимость)?
2. **Канбан-жест**: разрешаем ли завершение этапа перетаскиванием, или канбан
   read-only обзор, а действия — только в «Мой цех»? (рекомендую: drag = только
   ready→in_progress→done в пределах колонки-цеха, без переносов между цехами)
3. Клиент в заказе: обязательное поле или опциональное? (рекомендую опциональное,
   у 100 импортированных заказов клиента нет)
