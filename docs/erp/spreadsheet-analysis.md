## 2. Ключевая доменная логика (извлечённая)

### Единица движения — ПОЗИЦИЯ заказа
Заказ (№ сделки Bitrix, ~5xxxx) дробится на позиции: вид изделия × цвет/вариант.
Примеры: заказ 50224 «Газпром» = 9 позиций (лонгсливы/футболки/жилетки × подразделения);
55530 = 5 позиций по цветам. Подтверждает ответ discovery №3 («делить на изделия»).

### Типы производства (маршрут позиции)
Из мастер-листа + `Маршруты`:
| Тип | Закуп | Закрой | Швейка | ВТО | Образцы |
|---|---|---|---|---|---|
| Без изделий | – | – | – | – | – |
| Готовое изделие | ✓ | – | – | – | – |
| Крой | ✓ | ✓ | – | – | – |
| Пошив | ✓ | ✓ | ✓ | ✓ | – |
| Образцы | ✓ | ✓ | ✓ | ✓ | ✓ |
| Подряд | ✓ | – | – | – | – |

### Нанесения — отдельная ось (комбинируются)
Вышивка / Шелкография / ДТФ / Термоперенос / Прочие (пришив нашивок) — любая комбинация,
на крое ИЛИ на готовом изделии. В швейке встречается «В ожидании нанесения» → нанесение
может идти ДО пошива (на крое) — порядок этапов зависит от заказа.

### Статусы готовности в цехе (вычисляются из зависимостей)
- `Не готов к работе` / `Ожидает крой` / `Ожидает нанесение` / `Ожидает пошив` / `Ожидает готовое изделие` — блокирован, причина = незавершённый предыдущий этап
- `Готов к работе` — «можно ставить в календарь»
- `Готово` — этап завершён

### Приоритет постановки в календарь
`1. Готов к постановке` → `2. Ближайший срок` → `3. Ожидает` → `4. Завершено`.
Правило из «Логики планирования»: сначала готовые, затем по ближайшему сроку клиента.

### Материалы — гейт для закроя
Ткань: план поставки / факт поставки (или «ткань со склада», «от поставщика», «без поставки»).
Фурнитура/бирки: что заказано, план, факт. Менеджер: связь сделал, но статусы не проставлены →
в итоговое планирование не подгружено. Это известная дыра, в ERP закрываем статусами прихода.

### Итог заказа
`Дата сдачи факт` + статус `Сдан вовремя / с опозданием / заранее` + `Отгружено`.
Плюс `Буфер дней` и `Риск` (заготовка).

### Мощности (шт/день) — для календаря
Закрой 800/650, Швейка 500/400, Шелкография 2000/1700, DTF 500/400, Вышивка 300/250
(мощность/целевая; резерв = разница).

## 3. Найденные ошибки и слабые места таблицы

1. **Сдвиги колонок и Excel-артефакты**: в «Дней до срока» попадают даты `1900-01-01`
   (формула вернула дату вместо числа) и даже номер заказа `51628` (лист `2_Ткань`, сдвиг).
2. **Ручные поля-хвосты** в `5_План_швейки` / `План_вышивка` — даты, «Подтвержден»,
   «В процессе», «Готово» дописаны справа без заголовков (менеджер предупреждал: добавит завтра).
   В ERP это станут нормальные поля: дата постановки, подтверждение цеха, факт-статус.
3. **Частичное выполнение только в комментариях**: «одна пачка вышивается, ожидаем 24 шт
   на переделку», «осталось 7 шт», «отгружен частично». Нужен учёт кол-ва:
   выполнено X из N по этапу (+ переделки как отдельная величина).
4. **Цвет/вариант зашит в название** позиции («(Голубые) …»). Должен быть полем.
5. **Типы «Крой» и «Подряд»** есть в справочнике маршрутов, но в мастер-листе не встречаются —
   уточнить, живые ли.
6. **Календарные планы пустые** — самое трудоёмкое руками. Именно это ERP должен
   автоматизировать в первую очередь (предложить раскладку по дням из мощностей и приоритетов).
7. **Материалы не влияют на статус готовности** (менеджер сам отметил) — в ERP статус
   «Готов к крою» должен вычисляться и из факта прихода ткани/фурнитуры.
8. **Дубли статусов заказа между листами** (швейка знает про нанесение и т.п.) — следствие
   формул; в ERP статусы этапов живут в одном месте, цеховые экраны — это фильтры.

## 4. Маппинг на модель ERP (наш формат)

### Сущности
```
orders            — № сделки Bitrix, название, менеджер, дата запуска, срок клиента,
                    буфер дней, дата сдачи факт, статус сдачи, отгрузка
order_items       — позиция: заказ, вид изделия, цвет/вариант, кол-во, тип производства,
                    комбинация нанесений, комментарий
item_stages       — этап позиции: (позиция × цех) + зависимости (после каких этапов),
                    статус (waiting/ready/in_progress/done/skipped),
                    qty_done / qty_rework, план-дата, факт-дата, причина ожидания (вычисл.)
materials         — ткань и фурнитура per заказ/позиция: что, источник (закупка/склад/
                    давальческая/без закупки), план прихода, факт прихода, статус
sections          — цеха (уже есть в ERP: 11 цехов)
section_capacity  — мощность шт/день, целевая загрузка (по цеху)
calendar_slots    — цех × дата × позиция × кол-во на день × ответственный ×
                    статус при планировании × факт/комментарий
risks             — тип, заказ, описание, ответственный, срок решения, статус
defects (брак)    — этап, позиция, кол-во, причина, решение (переделка/списание)
weekly_commits    — планерка: заказ × цех × подтверждение × итоговый коммит
```

### Вычисляемая логика (была формулами — станет кодом)
- Статус готовности этапа = все этапы-зависимости `done` И материалы пришли
- Причина ожидания = первая незакрытая зависимость («Крой ещё не завершён»…)
- Приоритет календаря = f(статус, дней до срока)
- Дней до срока = срок клиента − сегодня; риск = f(остаток дней, буфер, объём)
- Загрузка цеха = Σ(кол-во в календаре на день) / мощность

### Экраны
| Экран | Из чего в таблице |
|---|---|
| Производственный план (мастер) | `1_Производственный_план` — все активные позиции, этапы-светофор, сроки, риск |
| План цеха (×8) | Цеховые листы — авто-фильтр: готов/ожидает+причина, приоритет |
| Календарь цеха | Календарные листы — раскладка по дням с учётом мощности, факт выполнения |
| Материалы (закуп) | `2_Ткань` + `Фурнитурабирки` — статусы прихода, гейтит закрой |
| Дашборд | Мощности + загрузка, просрочки, риски, узкие места |
| Планерка недели | `10_Планерка_недели` — коммиты цехов |
