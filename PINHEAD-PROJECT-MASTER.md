## 1. Что такое Pinhead Order Studio

**Pinhead Order Studio** — внутренняя система приёма заказов на пошив одежды с нанесением (шелкография, flex, DTG, вышивка, DTF).

**Кто использует сейчас:** менеджеры компании Pinhead.  
**Кто будет использовать:** покупатели напрямую (покупательский портал — будущий этап).

**Ключевой флоу:**
```
Выбор изделия → Ткань → Цвет → Крой → Размеры
→ Обработки (фурнитура)
→ Зоны нанесения + Техника (5 видов) + Бирки
→ Список позиций (multi-SKU)
→ Данные клиента
→ Итог + Сохранение в CRM
```

**Технологический стек (финальный):**
- React 19 + Vite 7 + Zustand 5
- Supabase (PostgreSQL + Auth + RLS)
- Vercel (деплой)
- Vitest + RTL + Playwright (тесты)
- CI/CD через GitHub Actions

## 2. Хронология версий и этапов

### 📁 ERA 1 — Vanilla HTML (до марта 2026)

**Файл:** `pinhead-order-studio-v1_7__5_.html`  
**Объём:** ~9 500 строк — один HTML файл со встроенным CSS и JS

#### Что было сделано:
Весь проект жил в одном `.html` файле. Supabase подключался через CDN-скрипт. Это давало максимальную скорость разработки — один файл, открыл в браузере, работает.

#### Версии в рамках эры:
| Версия | Что появилось |
|--------|---------------|
| v1.0 | Базовый конфигуратор: SKU → ткань → цвет → размеры → зоны → итог |
| v1.1–1.3 | Техники нанесения (5 видов), бирки, обработки |
| v1.4–1.5 | Мульти-позиции (items), история заказов, Kanban |
| v1.6 | Редактор цен, экспресс-калькулятор, ролевая модель (6 ролей) |
| **v1.7** | **Финальная версия эры:** аналитика (Recharts), Admin Panel, QR-коды, Bitrix поле, аудит-лог |

**v1.7, итерация 5** — последний зафиксированный файл этой эры. На этом момент vanilla-версия была функционально полной для внутреннего использования.

#### Почему решили уйти от vanilla:
- Файл стал неподдерживаемым (~9 500 строк в одном файле)
- Нет тестов — любое изменение могло сломать что угодно
- Нет типов — рефакторинг опасен
- Сложно добавлять новый функционал
- Невозможно масштабировать на покупательский портал

## 6. Почему проект застрял

### Главная причина — «Паралич планирования»

После mega session (20 марта) был создан грандиозный план на 5 фаз, включающий:
- TypeScript миграцию всех 40 файлов
- Полный рефакторинг Zustand-сторов
- Переход на CSS Modules
- И ещё 20+ задач

**Это слишком много для начала.** Проект завис потому что непонятно с чего начать, а начинать с TypeScript-миграции (самое скучное) не хочется.

### Вторичные причины

1. **Нет покупательского портала** — оригинальная цель ("для покупателей") до сих пор не достигнута
2. **Vanilla HTML живёт параллельно** — v1.7 файл существует как "запасной аэродром", что снижает срочность
3. **Технический долг пугает** — 47% без тестов + монолитный стор + нет TypeScript = страшно что-то трогать

## 7. Мозговой штурм

### Вопрос: Что нужно Pinhead прямо сейчас?

**Вариант A — «Стабилизация»**  
Доделать незакрытые технические задачи из плана 21.03. Убрать технический долг.  
*Плюсы:* надёжная база для дальнейшего  
*Минусы:* месяц работы, который клиент не видит

**Вариант B — «Покупательский портал»**  
Сразу строить фронт для покупателей. Отдельный домен/роут, упрощённый флоу.  
*Плюсы:* реальная ценность для бизнеса  
*Минусы:* текущий код с техдолгом станет сложнее поддерживать

**Вариант C — «Гибрид»**  
Минимальная стабилизация (Error Boundary + загрузка артворка + price breakdown) → покупательский портал.  
*Плюсы:* баланс между надёжностью и новым функционалом  
*Минусы:* требует чёткой приоритизации

### Идеи для покупательского портала

1. **Упрощённый визард** — только самые востребованные изделия, без экспертных параметров
2. **«Конфигуратор за 3 шага»** — изделие + нанесение (текст/лого) + размеры, без технических деталей шелкографии
3. **Онлайн-одобрение макета** — клиент видит мокап и нажимает «Одобрить»
4. **Трекинг статуса** — клиент видит где его заказ без звонков менеджеру
5. **Личный кабинет покупателя** — история заказов, повторный заказ

### Идеи для улучшения внутренней системы

1. **Загрузка артворка** — прикрепить файл к зоне нанесения (Supabase Storage)
2. **Price breakdown** — «Базовая цена 450₽ + шелкография 120₽ + срочность 50₽ = 620₽»
3. **Уведомления в Telegram** — бот сообщает менеджеру о новом заказе
4. **Bitrix синхронизация** — автоматическая передача заказа в CRM
5. **Мобильная версия Kanban** — менеджеры смотрят статусы с телефона

### 🛒 ЭТАП 2 — Покупательский портал MVP (2–3 недели)

Отдельный роут `/order` с упрощённым флоу для клиентов:

**Флоу покупателя (5 шагов вместо 6):**
1. Выбор изделия (только популярные SKU, карточки с фото)
2. Цвет + размеры (визуальный свотч, простая таблица)
3. Нанесение (загрузить файл / написать текст) — без выбора техники
4. Данные для связи
5. Итог + отправить заявку

**Что получает бизнес:**
- Клиент сам заполняет заявку → менеджер только проверяет
- Заявка сразу попадает в Kanban со статусом `review`
- Клиент получает ссылку для отслеживания статуса

**Технически:**
- Новый маршрут `/order` без авторизации (или по invite-ссылке)
- Переиспользование существующих компонентов
- Supabase Storage для артворков
- Email/Telegram подтверждение клиенту
